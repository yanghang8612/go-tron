## Real divergences found

### RED-1: `allow_new_reward` weight-increment branch in V1 freeze/unfreeze

- **java**: `FreezeBalanceActuator.java:136`, `UnfreezeBalanceActuator.java:243`
  use `dynamicStore.allowNewReward() ? increment : frozenBalance/TRX_PRECISION`
  where `increment = (newWeight - oldWeight)` post-mutation.
- **gtron**: `actuator/freeze_balance.go:117` always passes
  `frozenBalance/trxPrecisionActuator`; no `allowNewReward()` branch.
- **Argument it's OK**: the inline comment at `freeze_balance.go:111-118`
  argues the two paths are equivalent because V1 freeze amounts and existing
  totals are always TRX-aligned. Verified: `Validate` rejects
  `FrozenBalance < 1_000_000` (line 44) and old totals are sums of past V1
  freezes which were all 1-TRX-aligned. So `(newTotal/TRX) - (oldTotal/TRX) =
  frozenBalance/TRX` exactly. **No bug**, but the inline argument depends on
  Validate's lower bound; if that's ever relaxed the parity will silently
  break. Tag as RED-1 only for monitoring.
- **Recommendation**: no fix; consider an actuator test that asserts the
  weight delta matches `allowNewReward()` semantics for a few stake sizes.
- **Impact**: none today.

### RED-2: `DelegateResourceActuator` lock period — wrong units + missing gate

- **java**: `DelegateResourceActuator.java:69,212-219,251-259`.
  `lockPeriod` is measured in **blocks** (multiplied by `BLOCK_PRODUCED_INTERVAL`
  = 3000ms to get duration). `getLockPeriod` clamps:
  - `supportMaxDelegateLockPeriod()` (props #78 & #70 both active): contract's
    `lockPeriod`, or `DELEGATE_PERIOD/INTERVAL = 86400` (= 3 days in blocks)
    when 0.
  - Else: forces 86400 blocks regardless of contract value.
- **gtron**: `actuator/delegate_resource.go:94-100`. Writes
  `dr.ExpireTimeForBandwidth = ctx.PrevBlockTime + c.LockPeriod` — treats
  `c.LockPeriod` as **milliseconds**. No `supportMaxDelegateLockPeriod` gate.
  Validate also doesn't enforce the `lockPeriod >= remainTime` invariant from
  java line 264.
- **Impact**: consensus divergence on every locked delegation tx — gtron's
  computed expire time differs from java's by a factor of 3000. Will fork
  any chain that includes a locked DelegateResource tx.
- **Recommendation**: fix is larger than 10 lines (touches Validate +
  Execute + needs unit conversion + the `validRemainTime` check + the
  `supportMaxDelegateLockPeriod` gate). Filed as backlog. Not in this PR.

### RED-3: `allow_strict_math` not honored in `ExchangeProcessor`

- **java**: `ExchangeProcessor` constructor takes `useStrictMath`; when true
  the Bancor math uses `StrictMathWrapper.pow` / `addExact` rather than
  ordinary `Math.pow`. Gate set in
  `ExchangeTransactionActuator.java:69,218` and read in
  `ContractStateCapsule.java:84` / `Program.java:2366`.
- **gtron**: `actuator/exchange_processor.go:29,39` uses Go `math.Pow`
  unconditionally. No `AllowStrictMath` branch.
- **Impact**: any post-#87 ExchangeTransaction whose ratio causes
  `Math.pow` vs `StrictMath.pow` to round differently will silently fork.
  Pre-#87 the two paths are identical by definition. **Proposal #87 is
  active on mainnet (passed in 2025).** This is a live consensus bug for
  any historical block that exercises an exchange tx after the activation
  height.
- **Recommendation**: not a surgical fix — requires implementing a Go
  equivalent of Java's `StrictMath.pow` (deterministic across JVMs by spec).
  Filed as backlog with potentially high priority depending on conformance
  replay coverage. Verify if conformance harness has hit this yet.

### RED-4: `consensus_logic_optimization` block-validity check is unconditional

- **java**: `DposService.java:121,135` — rejects blocks with
  `timestamp % BLOCK_PRODUCED_INTERVAL != 0` and rejects `slot == 0` only
  when `allowConsensusLogicOptimization()` is true (proposal #88).
- **gtron**: `consensus/dpos/verify.go:38` always rejects mod-mismatch
  timestamps; no slot==0 guard exists at all.
- **Impact**: in practice, mainnet blocks are always 3000-ms-aligned because
  java's producer code only mints aligned blocks. So gtron's stricter check
  is functionally identical to post-#88 java. But if any historical block
  on the mainnet header chain has a misaligned timestamp (e.g. early
  history before #88 activated), gtron will reject it while java accepts.
  Practical impact: low. Theoretical fork-divergence: real.
- **Recommendation**: gate behind
  `ctx.DynProps.ConsensusLogicOptimization()` to match java; also add the
  `slot != 0` guard inside the same gate. Surgical (~6 lines) — but
  consensus-critical, so I'd want a unit test that exercises both
  branches before landing. Filed as backlog; out of scope for the
  no-touch-mainnet rule this pass.
