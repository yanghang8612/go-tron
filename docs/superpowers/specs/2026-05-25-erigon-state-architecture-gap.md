# Erigon-Style State Architecture Target And Gap

Date: 2026-05-25

Status: active architecture gap document

Scope: fresh database only. No migration path is required. The target is to make
gtron's state/storage architecture converge on Erigon's domain model while
preserving java-tron wire compatibility and consensus behavior.

## Non-Negotiable Constraints

- java-tron remains the consensus source of truth: block bytes, transaction
  bytes, protobuf encoding, P2P behavior, actuator semantics, fork gates, and
  DPoS behavior must not change.
- The refactor may delete incompatible old local storage formats. Existing DB
  migration is out of scope.
- TRON's externally visible `accountStateRoot` / state-root compatibility
  boundary must be explicitly preserved or intentionally version-gated. Any
  internal root redesign must not silently change java-tron-compatible block
  validation behavior.
- Hot latest data and cold archive data must be physically separated. Archive
  queries must not require keeping all history in Pebble hot KV.

## Target Architecture

The desired end state is an Erigon-style domain system:

```text
Block execution
  -> SharedDomains / DomainTx
      -> hot latest domain tables
      -> hot history changesets and inverted indexes
      -> touched-key set for commitment
      -> immutable snapshot files
      -> pruning watermark
  -> CommitmentDomain
      -> internal root / block state root adapter
```

The physical domains should be explicit and uniform:

| Domain | Purpose | Key shape |
| --- | --- | --- |
| `AccountsDomain` | canonical account envelope | `account_id20` |
| `AccountKVDomain` | account-owned logical KV, including contract storage and system stores | `owner20 || generation_u64 || domain_u16 || logical_key` |
| `KVGenerationDomain` | per-account storage generation | `owner20` |
| `CodeDomain` | immutable bytecode by hash | `code_hash32` |
| `CommitmentDomain` | commitment branch/node/checkpoint state | commitment-specific logical key |
| `HistoryDomain` | pre-values and inverted/accessor metadata | `txNum`, logical key, file offset |

The hot store should keep current/latest state plus recent changes. The cold
store should hold immutable files:

```text
latest:
  *.seg   latest key/value records
  *.lidx  latest ordered offset accessor

history:
  *.seg   state-domain-change records
  *.idx   txNum -> record offset index
  *.kv    logical-key -> record offset accessor

manifest:
  visible txNum range
  generation
  active segments
  retired segments
```

Archive reads should resolve values by key:

1. Start from latest value.
2. Walk hot changes through inverse indexes, not global block/tx range scans.
3. Walk cold history through key accessors, not full segment scans.
4. Merge only the exact logical key families required by the query.

## Current State In The Tree

The current working tree has already moved a large part of the state engine in
this direction.

### Flat Latest And Temporal Domains

- Flat latest account rows, account-KV rows, and KV generation rows exist under
  `core/rawdb`.
- `StateDomainChange` is the canonical temporal mutation record for account
  latest, KV latest, and KV generation:
  `core/rawdb/accessors_state_changeset.go`.
- StateDB now captures flat-domain changes during block execution:
  `core/state/domain_change_journal.go`, `core/state/statedb.go`.

### Keyed Archive Reads

- `PersistentHistoryReader` reconstructs account, code, and storage from flat
  state-domain history:
  `core/state/history.go`.
- Keyed reads use hot inverse indexes first:
  `collectStateDomainChangesByKey`,
  `HistoryDomain.IterateHotHistoryChanges`.
- Cold history supports keyed access through:
  `StateDomainChangeColdKeyHistory`.
- `readStateAccountKVAsOf` now merges exact `KVLatest` and `KVGeneration`
  keys instead of scanning all cold history.

### State-Domain Cold History

- Binary state-domain history files are written as a required trio:
  `.seg + .idx + .kv`.
- Manifest validation rejects incomplete binary history trios:
  `validateHistoryBinaryCompanionTriples` in
  `core/state/snapshots/manifest.go`.
- Binary range reads require `.idx`; keyed reads require `.kv`:
  `core/state/snapshots/history_segment.go`.
- Cold state-domain-change segments carry block-to-txNum range metadata for
  archive readers, and the backend/reader can recover `StateTxRange` from cold
  snapshots after hot history rows are pruned.
- The pruning checker validates history `.seg`, `.idx`, and `.kv` files through
  registered file-native checkers:
  `core/state/pruning/checker.go`.

### Latest Snapshot Files

- Latest snapshots are binary `.seg` files when generated by the aggregator:
  `core/state/snapshots/latest_binary.go`.
- Latest snapshots now have `.lidx` ordered offset accessors.
- Latest snapshots also have `.bt` BTree-style separator accessors for point
  lookup and prefix seek; managers prefer `.bt`, then `.lidx`, then legacy
  segment reads.
- `Manager.Get*` and prefix iteration use `.bt` / `.lidx` plus `ReadAt` when
  available, avoiding full latest segment materialization:
  `core/state/snapshots/latest_segment.go`.
- The aggregator emits latest `.lidx` and `.bt` companions for every latest
  `.seg`:
  `core/state/snapshots/aggregator.go`.
- The pruning checker validates latest accessors:
  `CheckLatestAccessorSegment`, `CheckLatestBTreeSegment`.
- Latest `.lidx` accessor checking is file-native: it streams checksum/size
  verification, reads the header, and validates offset bounds/order with
  `ReadAt` instead of materializing the full accessor table.
- Snapshot segment checks are dispatched through `DomainRegistry` so latest,
  history, index, accessor, and BTree validation stay attached to registered
  domains.
- History open/write/range-read/key-read operations are dispatched through
  `DomainRegistry` codec descriptors for the registered history family.
- The pruning checker validates `CodeHash` retention for latest account
  envelopes and hot/cold state-domain history against hot `state-code-v1` rows
  or cold CodeDomain snapshots.

### Snapshot Lifecycle

- Snapshot manifests track visible tx range, generation, active segments, and
  retired segments:
  `core/state/snapshots/manifest.go`.
- The cold builder publishes registry-declared state-domain-change history
  files:
  `core/state/snapshots/cold_builder.go`.
- The compactor selects history families through `DomainRegistry` and merges
  binary state-domain-change history trios:
  `core/state/snapshots/compactor.go`.
- Hot history pruning is gated by registry-derived snapshot coverage:
  `core/state/pruning/worker.go`.

## Remaining Differences From Erigon

The following items are the remaining architecture gaps. They are listed in the
order that most directly reduces DB size and archive-read cost.

### 1. Latest Writers Must Stay Streaming

Older builders collected all latest entries in memory before writing a segment:

- `BuildAccountLatestSegmentFromDB`
- `BuildLatestDomainSegmentFromDB`
- `BuildKVGenerationSegmentFromDB`
- `BuildCodeSegmentFromDB`
- `BuildCommitmentRootSegmentFromDB`
- `BuildCommitmentNodeSegmentFromDB`

That shape is not Erigon's streaming file builder model.

Status update:

- The production binary `.seg` aggregator path now streams latest entries from
  rawdb iterators and emits `.seg + .lidx + .bt` in one pass.
- The public DB builders listed above now route through
  `writeLatestSegmentFromIterator`; they no longer allocate a resident
  `[]LatestEntry` for either binary `.seg` output or legacy JSON output.
- The legacy JSON writer is also streaming and enforces strictly sorted input.
  `WriteLatestSegment` remains available for tests/compatibility callers that
  already have an in-memory entry slice.
- AccountLatest, AccountKVDomain, KVGeneration, CodeDomain, and CommitmentDomain
  latest snapshot build/restore now use a typed `latestHotStore` boundary.
  rawdb iteration/read/write functions are confined to `rawDBLatestHotStore`,
  while the JSON/binary segment writers and restore switch consume typed rows.
- Production binary latest restore now streams entries directly from `.seg` into
  `latestHotStore` after streaming checksum/size verification. It no longer
  calls `OpenLatestSegment`/`readLatestBinarySegment` or populates the manager's
  materialized segment cache for binary restore.
- Production latest snapshot checking now uses a file-native binary `.seg`
  checker. `CheckRegisteredSegment` validates binary latest header metadata,
  checksum/size, sorted keys, and dataset-specific key/value rules without
  materializing the segment through `OpenLatestSegment`.
- Production latest accessor checking now uses a file-native `.lidx` checker.
  It validates accessor checksum/size, header metadata, referenced segment
  size, and strictly increasing record offsets without calling
  `readLatestBinaryAccessor`.

Next step:

- Keep production latest snapshot generation on the streaming binary writers,
  and avoid adding new latest builders that accept resident entry slices.
- Keep `WriteLatestSegment` and `OpenLatestSegment` only for tests and legacy
  JSON fallback.

Acceptance:

- Building latest snapshots does not require memory proportional to total latest
  state size.
- A large latest snapshot build, restore, or check has peak memory bounded by
  iterator buffers and a small offset buffer or streamed accessor writer.

### 2. Latest Accessor Is Ordered Offsets, Not Full Erigon BTree/Recsplit

The `.lidx` file stores ordered record offsets and binary-searches by reading
keys from `.seg`. This is intentionally simple and file-native, but it is not
yet equivalent to Erigon's `.bt`, `.kvi`, or recsplit-backed accessors.

Status update:

- Latest binary snapshots now include `.bt` separator files. Point lookup first
  seeks through `.bt`, then scans only the bounded leaf block in `.seg`.
- Prefix iteration uses `.bt` to seek to the first candidate block instead of
  scanning from the beginning of the segment.
- `.lidx` remains as a stable ordered-offset fallback.

Next step:

- Recsplit/hash accessor evaluation CONCLUDED (2026-05-27 audit): not adopted.
  `Manager.getLatestValueFromRef`/`iterateLatestPrefix` prefer `.bt` (bounded
  leaf-block scan for point lookup, BTree floor seek for prefix), then `.lidx`,
  then a full scan only when no companion exists — so both acceptance criteria
  are already met. A hash-only recsplit index cannot replace this because gtron
  requires ordered cursors for `IterateKVLatestPrefix` and
  `IterateCommitmentNodes`; `.bt` provides ordered seek that a hash index would
  not. Revisit only if a profile shows `.bt` point-lookup cost dominates.

Acceptance (met):

- Point lookup does not require `O(log n)` disk key reads for every query.
- Prefix iteration can seek to the first matching key without scanning from the
  beginning of the segment.

### 3. `StateDomainChange` Cold Files Need File-Native Production Paths

History `.seg + .idx + .kv` is now required for production cold history.
Legacy/test helpers can still materialize whole files, but runtime archive
reads and pruning checks should use file-native indexes or mmap-friendly
structures.

Status update:

- Production keyed archive reads use file-native `.kv` offset tables and
  `ReadAt`; they no longer allocate the full accessor.
- Production tx-range history reads use file-native `.idx` binary search and
  `ReadAt`; they no longer allocate the full index.
- Runtime cold-history iteration is registered as streaming callbacks. Manager
  range and keyed iteration now dispatch through `IterateHistoryRange` /
  `IterateHistoryByKey`, so callers can stop early without first materializing
  all matching changes from a cold segment.
- Cold block-to-state-tx-range lookup now uses the binary `.idx` file to
  lower-bound by block and `ReadAt` only the candidate state-domain-change
  records. `Manager.StateTxRangeForBlock` no longer opens and decodes an entire
  history segment to derive a block's txNum range.
- The checker validates `.idx` and `.kv` by streaming/random reads. Full
  accessor decoding remains as a test/debug helper.
- The checker validates binary history `.seg` files by streaming record frames
  from disk. `CheckRegisteredSegment` no longer calls
  `OpenStateDomainChangeSegment` for binary history, so pruning checks validate
  checksum/size, header metadata, tx ordering, and trailing bytes without
  materializing all changes.
- File-native history readers bound record and accessor-frame payload lengths
  by the actual file size before allocation, so corrupt cold files cannot force
  large transient buffers during range, block-range, keyed, or checker reads.
- Production history compaction now uses a file-native state-domain-change
  compactor. It copies binary `.seg` payloads, rewrites `.idx` offsets, and
  multiway-merges sorted `.kv` accessors while validating companion entries
  against the source records, rather than opening every candidate segment into
  a resident changes slice.
- Runtime snapshot managers and pruning checks now load production manifests
  through a stricter manifest gate: registered history domains must publish the
  binary `.seg + .idx + .kv` trio. JSON state-domain-change history remains
  available only as a direct legacy/test segment reader or writer, not as a
  production cold-history manifest format.

Next step:

- Prefer a two-level ordered index or recsplit-like accessor that returns record
  offsets directly.
- Keep any new production history domain on binary segment/accessor/index files
  from the start.

Acceptance:

- Keyed archive reads do not allocate memory proportional to the number of
  history changes in a segment.
- Missing binary companion files are fatal for production manifests.

### 4. Domain Registry Is Only Partially Applied

`DomainRegistry` exists and now describes registered latest/history families,
path stems, companion requirements, builders, codec operations, and progress
semantics. The aggregator iterates registry configs to build latest/history
files, manifest validation enforces registry-declared binary latest and history
companions, and the pruning checker uses registry dispatch to validate
registered segment families.

Erigon's model is centered on registered domains with uniform latest/history/
index/accessor lifecycle.

Remaining gap:

- State-domain-change now declares open/write/range-read/key-read codec
  operations in the registry, and compactor / cold readers consume those
  operations through the registered config.
- History compaction now has a generic `CompactHistoryDomain` entry point and a
  `HistoryCompactionResult` carrying the registered dataset and emitted segment
  refs. `HistoryDomain` also registers a binary streaming compactor for the
  state-domain-change file trio.
- The cold history builder now targets a configured `HistoryDataset`, validates
  that it is a registered history domain, and reports emitted history/accessor/
  index refs rather than assuming a single state-domain-change file.
- The only registered history codec is still `state-domain-change`, so the
  lifecycle is proven for one history dataset per runner rather than for a
  multi-domain aggregation step.
- Adding a new history domain still requires implementing its codec/builder and
  choosing whether it needs a distinct progress/merge policy.

Next step:

- Add another history domain only when the execution model actually produces a
  second temporal family; otherwise keep CodeDomain content-addressed.
- Promote cold builder/pruner/compactor scheduling into an explicit multi-domain
  aggregation step if a second temporal family is introduced.

Acceptance:

- Adding `CodeDomain` or another system domain requires registering config, not
  editing every builder/checker/manifest switch.
- Latest/history/accessor naming and validation are generated from domain
  config.

### 5. Code Domain Retention Still Needs A Final Policy

Code is content-addressed in rawdb, account envelopes reference `CodeHash`, and
`CodeDomain` is now a registered latest snapshot family with `.seg + .lidx +
.bt` files. Snapshot restore writes code bytes back into the content-addressed
rawdb code domain, and archive `CodeAt` can fall back to cold CodeDomain latest
snapshots when hot code bytes are absent.
The pruning checker rejects latest account rows and account-envelope history
changes whose `CodeHash` is not backed by either hot code bytes or a CodeDomain
snapshot covering the referenced txNum.
Live `StateDB.GetCode` can fall back to content-addressed CodeDomain snapshots
at or before the active state txNum, so snap-mode pruning can delete hot
`state-code-v1` rows once all account-envelope references to the hash are backed
by CodeDomain snapshots.
Execution-facing code reads and writes are now behind a typed `stateCodeStore`
boundary in `core/state`; rawdb remains the default content-addressed
implementation, but `StateDB` no longer calls `ReadStateCode` /
`WriteStateCode` directly from `GetCode` or commit.
Coverage now exercises archive `CodeAt` with actual hot code deletion and live
TVM execution after `state-code-v1` pruning.

Remaining gap:

- Code has no temporal changeset because account-envelope history selects code
  by hash. Hot code now has a CodeDomain coverage-gated prune step, but the
  policy still relies on content-addressed CodeDomain snapshots instead of a
  distinct temporal code family.

Next step:

- **Done (2026-05-27):** CodeDomain is documented as the intended final
  content-addressed latest-retention policy (plus account-envelope temporal
  selection) — see the docstring on the `CodeDomain` registry entry in
  `core/state/snapshots/domain_registry.go`. The content-addressed model has a
  behavioral anchor (`TestPersistentHistoryReaderReadsCodeFromColdCodeDomain`
  proves the cold CodeDomain retains a superseded bytecode version, so historical
  `CodeAt` across an in-place overwrite is served by account-envelope history
  selecting the right hash plus content-addressed retention of every referenced
  hash), and the snapshot-coverage deletion gate is now locked from the negative
  direction by `TestWorkerSnapPreservesHotCodeWithoutCodeDomainCoverage`
  (uncovered hot code is never pruned). No separate temporal code changeset is
  needed.
- Add java-tron parity fixtures for historical contract code if future
  compatibility work finds a case where content-addressed retention is
  insufficient.

Acceptance:

- Contract bytecode can be restored from snapshots or read directly from
  CodeDomain snapshots without relying on unrelated legacy flat paths.
- Archive `CodeAt` semantics are defined by account-envelope history plus
  content-addressed code retention.

### 6. CommitmentDomain Is Not Erigon's Commitment Pipeline

The current implementation has a working commitment/root layer. Generic
account-KV writes now flow through `SharedDomainTx` with a commitment mutation
recorder, so prefix deletes and exact puts/deletes are collected as logical
domain touches before the latest table is flushed. `StateDB.Commit` also records
account latest and KV generation touches through the same
`DomainCommitmentState`, then derives all flat-domain commitment updates from
that recorder after the latest tables are flushed. Commitment root, branch-node,
and checkpoint rows are stored in the physical `CommitmentDomain` keyspace.
When a fresh database has no stored commitment root yet, the first commit can
bootstrap the root from the collected logical touches instead of rebuilding by
scanning every latest table. Snapshot aggregation includes commitment root,
branch-node, and checkpoint families so cold restore can reconstruct the
persisted `CommitmentDomain` state. The adapter between go-tron's internal
CommitmentDomain root and java-tron's block-header `accountStateRoot` is now
explicit in `core.StateRootAdapter`: block production and replay compute the
java-compatible root through the adapter, while the internal root is persisted
only out-of-band through `rawdb.WriteBlockStateRoot`. Latest-mode tests pin that
a valid block's java `accountStateRoot` can differ from the internal
CommitmentDomain root, and that a mismatched java root is still rejected before
the internal state is committed.
If the latest-root row is missing but CommitmentDomain branch nodes are still
present, the root row is repaired from the persisted root node before applying
logical touched-key updates. That keeps normal root-row repair off the
full-latest rebuild path when branch state exists.
The latest-root apply/repair path and commitment checkpoint seek are now behind
the `domains.ApplyLatestCommitment` / `domains.SeekLatestCommitment` boundary.
`StateDB`, `DomainCommitmentState`, and `FlatStore` call that typed domain
adapter instead of open-coding rawdb root, branch-node repair, rebuild, update,
or checkpoint iteration logic at their execution sites. The commitment helper
itself now consumes a typed `latestCommitmentStore`; rawdb commitment
root/repair/rebuild/update/checkpoint accessors are confined to the
`rawDBLatestCommitmentStore` compatibility adapter.
Commitment checkpoint writes also maintain a `latest-checkpoint` pointer in the
CommitmentDomain keyspace. `domains.SeekLatestCommitment` reads that pointer
through the typed store before falling back to checkpoint iteration, so normal
restart/commitment seek is O(1) instead of scanning all retained checkpoints.
The CommitmentCheckpoint snapshot family now carries that pointer as part of
its registered latest segment. Snapshot build streams the checkpoint rows and
synthesizes the pointer from the highest checkpoint when the hot pointer is
missing, so snapshot restore preserves the O(1) seek path instead of restoring
into the fallback scan path.
Commitment apply now treats the latest-root row and branch-node availability as
separate invariants. A no-op apply returns the persisted latest-root row without
reading branch nodes, so pruned branch state cannot accidentally rewrite the
root to zero. Non-empty applies verify that the persisted root node exists and
matches the latest-root row before applying touched-key updates; if branch state
has been pruned, the typed commitment store rebuilds branch state before the
update instead of computing a new root with missing sibling branches.
`StateDB` can now pass the cold CommitmentDomain snapshot manager into
`domains.ApplyLatestCommitmentWithSnapshotRepair`. When the hot latest-root row
or latest checkpoint identifies a root that matches the cold CommitmentRoot
snapshot, the typed commitment store restores branch nodes from the registered
CommitmentNode snapshot before applying touched-key updates. The full-latest
rebuild path is now reserved for stores without a matching cold branch snapshot
or with stale/corrupt snapshot data.

Status update (2026-05-27):

- An Erigon-style staged commitment engine now exists behind the
  `DatabaseConfig.StagedCommitment` gate (now the default — see the flip note
  below): prefix-keyed `BranchData`
  nodes in a dedicated `state-commitment-branch-v1-` keyspace, a hex-patricia
  fold engine (`core/state/domains/commitment_tree.go`, fuzz-proven that
  incremental folds equal a from-scratch recompute), wired through the existing
  `LatestCommitmentStore` seam via `ApplyLatestCommitmentWithStore[AndRepair]`
  and selected at commit time by `StateDB.stagedCommitment()`.
- A normal commit folds incrementally over persisted branch state and does NOT
  trigger a full-latest rebuild (regression-guarded by the staged-engine
  "no bootstrap on normal commit" test).
- Fork-switch/rewind is correct for the staged engine: the block buffer discards
  orphan-branch layers (including their branch + root rows) keyspace-agnostically,
  so replay folds onto LCA-restored branches; `stagedCommitmentStore.Rebuild`
  clears stale branches before re-folding. A gate=true fork-switch test asserts
  the post-switch root matches the head state root and a from-scratch staged
  rebuild.
- The internal staged root stays decoupled from java-tron's `accountStateRoot`
  (via `core.StateRootAdapter`), so the engine choice is consensus-safe regardless
  of which one runs.
- Default flipped to staged 2026-05-27: `NewDatabase` now constructs the staged
  engine, and the `StagedCommitment` gate has since been removed entirely (staged
  is the only engine). Within-engine equivalence oracles (incremental commit ==
  from-scratch staged rebuild) were repointed off the legacy
  `RebuildLatestDomainCommitment`; full `go test ./...` is green.

Remaining gap:

- Staged cold restore is a full production path as of 2026-05-27 (commits
  `87298e3`→`0ddb73f`): `RestoreNodesFromSnapshot` restores branch rows from a
  `CommitmentBranchSnapshotSource` and self-verifies by re-folding to the expected
  root, degrading to `Rebuild` only when no snapshot matches. The production
  latest-snapshot driver builds + publishes `CommitmentBranch` segments and the
  snapshot `Manager` implements `CommitmentBranchSnapshotSource` (resolving the
  covering branch ref against the current manifest per call), so the bare `Manager`
  wired into the chain drives restore. An e2e test proves restore folds from the
  snapshot with no full-scan rebuild and yields the same root as the rebuild path
  (see #7 for the latest-build driver).
- Incremental rewind without rebuild — slice 1 landed 2026-05-27: an inverse-delta
  unwind primitive (`rawdb.CollectStateUnwind` + `domains.UnwindCommitment`) rewinds
  the latest-domain tables and staged commitment branches from the tip to a target
  block by folding the INVERSE of the persisted `StateDomainChange` pre-images
  (`Prev`/`Next`, within `HistoryPruneWindow`), touching only changed branch paths —
  no full `Rebuild` scan, no new persistence. A property test proves rewind-to-N
  reproduces the from-scratch branch keyspace BYTE-for-byte (12 rows incl.
  intermediate branch create/collapse), the same root, and that forward commits from
  the rewound state match a from-scratch store. The unwind self-verifies the re-folded
  root against the target block's persisted internal root, so an incomplete/pruned
  range errors instead of corrupting. Also fixed a latent restart-sync bug:
  `ResetMutableState` now clears the commitment branch keyspace (it previously kept
  stale branches that could let `RestoreRootFromNodes` skip `Rebuild` with the prior
  run's root). Slice 2 (still open) wires a consumer (`BlockChain.UnwindTo`); until
  then the "rewind without rebuild" acceptance is met only vacuously — no current path
  rewinds into the old Rebuild fallback (fork-switch uses the buffer; restart-sync
  replays from genesis).
- Legacy-engine removal (Phase 2) — DONE 2026-05-27 (commits `bb99cb6`→`e9abd5c`,
  full `go test ./...` green). The legacy binary-radix engine is entirely gone: the
  store + db-wrappers, the `StagedCommitment` gate, the dead `UnwindStateDomainChanges`
  path, all bit-tree rawdb helpers (`Rebuild/UpdateLatestDomainCommitment`,
  `ComputeAndWriteLatestDomainRoot`, `tree/node` prefix), the `CommitmentNode`
  (`tree/node/`) snapshot family across `snapshots/{manifest,latest_segment,
  latest_binary,domain_registry}.go`, and the `latestDomainCommitmentStore` interface.
  The pruning checker iterates the commitment domain directly
  (`rawdb.IterateStateCommitmentDomain`) for the engine-agnostic root + checkpoint
  validation, and additionally counts published `CommitmentBranch` snapshot rows
  and warns on staleness; staged branch rows are derived state and are never pruned.
- Snapshot-streaming production path (DONE 2026-05-27): the aggregator builds +
  the `Manager` serves staged `CommitmentBranch` snapshots in production. Operator
  note: latest snapshots are built from the live DB without a consistent read
  snapshot, so a build can capture a cross-dataset-inconsistent dump — self-detected
  via the restore re-fold/expected-root check (falls back to `Rebuild`), so it never
  corrupts state. Inherent to the latest-snapshot design (shared by all latest datasets).
- Expand java-tron fixture coverage around root-relevant blocks and contracts.

Acceptance:

- Recomputing root from latest domains and commitment state is deterministic.
- Rewind/unwind can restore commitment state without rebuilding the entire
  latest state.
- Java-tron compatibility tests cover root-relevant blocks and contracts.

### 7. Cold/Hot Lifecycle Is Not A Full Step-Based Aggregator

The current cold builder and pruning worker are separate pieces. Erigon uses a
step-based aggregator lifecycle: collate, build files, publish visible view,
prune hot data covered by files, compact old files, and maintain progress
watermarks.

Status update:

- Manifest progress tracks latest, history, accessor, commitment-flush, and
  hot-prune watermarks.
- Cold builder validates history progress against visible
  state-domain-change coverage so a restart cannot silently skip uncovered
  history.
- The cold builder writes `SnapshotBuild` stage progress when it publishes a
  new history segment.
- `SnapshotPrune` stage progress is now advanced by the pruner, not by
  `InsertBlock`.
- Manifest latest/history/accessor/commitment-flush/hot-prune txNum watermarks
  are mirrored through the snapshot stage-progress adapter whenever the
  builder, aggregator, or pruner has a DB writer, keeping rawdb stage writes at
  the package boundary.
- The node now registers one ordered snapshot/prune lifecycle that runs cold
  history build/publish/compaction before hot pruning, so snap mode cannot prune
  history rows before the same pass has made snapshot coverage visible.
- The lifecycle now also runs a production LATEST-snapshot build pass (landed
  2026-05-27, commits `87298e3`→`0ddb73f`): `Aggregator.BuildLatest` builds every
  registered latest dataset (accounts, KV, KV-generation, code,
  commitment-root/checkpoint, and `CommitmentBranch`) at the solidified head,
  integrates them, and writes latest/commitment-flush stage progress. It is gated
  by the `LatestBuildBlocks` cadence knob (default ~33h of blocks; all latest
  datasets share one cadence), seeded to the current solidified block on start to
  skip a startup full-scan. This closes the gap where the latest-build path
  (`Aggregator.Build`) had no production caller — only the history pass ran.
  Latest-build watermark persistence landed 2026-05-27 (commit `42f08d4`): a
  forward-only `StageSnapshotLatestBuild` block stage is written after each latest
  build and seeded on `Runner.Start`, so a restart resumes the cadence gate instead
  of re-seeding to the current head (fresh nodes still fall back to the head, so no
  startup full-scan / no migration). The step-aggregator capabilities (collate,
  build, publish, prune, compact, watermarks) already run in order under
  `SnapshotLifecycle.OnePass` with shared manifest + `StageSnapshot*` watermarks, so
  #7's Acceptance (resume without guessing; retention explainable by watermarks) is
  met. Consolidating `Runner`+`Pruner` into a single "aggregator" type was
  intentionally skipped — Acceptance is behavioral, not a class layout, and merging
  the two clean concerns (build vs. delete) is churn with no behavior change.
- Canonical execution stage progress is written through a typed
  `stageProgressStore` boundary and restart-sync rewind explicitly rewinds
  `Headers`, `Bodies`, `Execution`, `Commitment`, and `Finish` stage progress to
  the target height.
- Fork switching rewinds canonical stage progress to the LCA before replaying
  the winning branch, then normal block application advances it to the new tip.

Next step:

- Extend the lifecycle to multiple registered history domains if/when a second
  temporal family is introduced.

Acceptance:

- A node can resume snapshot building/pruning after restart without guessing.
- Hot state/history retention is explainable by watermarks and manifest
  coverage.

### 8. Execution Is Still Single-Block `InsertBlock`, Not Staged Sync

The chain still executes mostly through `BlockChain.InsertBlock` and per-block
StateDB commit. Erigon's model batches execution in stages and flushes domains
by stage progress.

Status update:

- Canonical stage progress is now advanced at the actual block-application
  phase boundaries instead of by one end-of-block helper: `Headers`/`Bodies`
  after header validation and block-layer creation, `Execution` after
  transaction/reward/statistics execution, `Commitment` after latest-domain
  commit and CommitmentDomain root update, and `Finish` after block metadata
  and hooks.
- Failed execution discards the active block layer, including any provisional
  stage progress, so a rejected block cannot move canonical stage watermarks.
- `BlockChain.InsertBlocks` provides a range-shaped staged import surface that
  holds the chain lock once while applying a fetched block range. Linear ranges
  now reuse one post-commit `StateDB` across successful blocks instead of
  reopening state from the parent root at every block boundary. It still commits
  each block independently; the remaining architectural work is to make this
  range share domain transactions and flush by stage.
- `StateDB.Commit` now has a reusable `CommitScope` carrying the
  `SharedDomainTx`, latest-domain writer, history reader, and commitment
  delegate used for generic account-KV writes. `InsertBlocks` and fork replay
  pass one scope across the applied range, so successful linear batches reuse
  the same domain transaction objects while still producing per-block roots and
  history ranges.
- History txNum allocation is now planned at the range/replay boundary.
  `InsertBlocks` and fork replay seed one `StateTxRange` allocator from the
  parent block, then hand each block an explicit planned range, instead of each
  `applyBlock` re-reading the parent txNum and deriving history time locally.
- Temporal state changes now publish through an explicit `DomainChangeStage`
  created from the planned range. Transaction-level journal deltas and
  block-final deltas are flushed through that stage, while the legacy
  StateDB-direct methods remain as compatibility shims for standalone
  `ProcessBlock` callers and tests.
- `StateDB` journal capture now separates temporal change construction from
  publication. The stage owns a `StateDomainChangePublisher` boundary and
  publishes the collected `StateDomainChange` rows through that interface, with
  rawdb as the current backing adapter rather than being called from the journal
  walker itself.
- The legacy commit-capture shim now uses the same metadata/seq preparation and
  publisher helper for account latest, account-KV latest, and KV generation
  changes. Production `core/state` execution code no longer calls
  `rawdb.WriteStateDomainChange` except through the rawdb publisher adapter.
- Hot temporal history publication is now split into explicit tx-range, row,
  and inverse-index writes. `StateDomainChangeRunner` composes those steps
  through the registered `HistoryDomain` config, making the hot history runner
  use the same domain registry that drives cold history build/open/read/check
  dispatch instead of hiding publication inside one rawdb accessor.
- Hot keyed history reads now also dispatch through the registered
  `HistoryDomain` config. `PersistentHistoryReader` asks the registry to
  iterate keyed hot changes for a tx window instead of switching directly over
  rawdb account/KV/generation index helpers or composing block scans itself.
- The lower-level rawdb account-latest, KV-latest, and KV-generation as-of
  helpers now reuse the same keyed hot-change iterator, so their compatibility
  path no longer duplicates inverse-index block collection and per-block change
  filtering logic.
- Prefix archive iteration now uses a matching hot prefix-change iterator, and
  account-KV single-key/prefix generation-reset paths collect KV changes plus
  generation changes through shared helpers instead of maintaining separate
  touched-block merge loops.
- The registered `HistoryDomain` now exposes the hot prefix-change iterator and
  account-KV prefix as-of entry point. `StateDB.IterateAccountKVAsOf*` resolves
  prefix archive reads through that registry config instead of directly calling
  the rawdb compatibility helper.
- The same registered `HistoryDomain` now exposes the account-KV single-key
  as-of reader. `StateDB.GetAccountKVAsOf*` converts block queries to txNum
  ranges and dispatches through the registry, leaving rawdb as the backing
  compatibility implementation.
- The registered `HistoryDomain` also exposes hot physical-domain as-of
  readers for account latest, account-KV latest at a concrete generation, and
  KV generation. These are intentionally registered as hot-domain hooks; archive
  readers that must merge cold history with hot tail changes still keep that
  merge boundary in `PersistentHistoryReader`.
- `HistoryDomain` now owns hot state tx-range publication, lookup, iteration,
  and deletion. `StateDB.Commit`, archive account-KV reads,
  `PersistentHistoryReader`, archive RPC availability checks, commitment seek,
  and code-retention pruning resolve tx-range lifecycle work through that
  registered hook; rawdb remains the backing compatibility implementation
  instead of an upper-layer dependency.
- Hot-only `PersistentHistoryReader` as-of reads now dispatch through the
  registered account-latest/KV-latest/KV-generation/account-KV hooks. The reader
  keeps its explicit merge path only when a cold history provider is attached,
  because that path must combine cold segment rows with the hot tail.
- Latest-domain hot point reads are now registered on the latest-domain configs
  themselves: AccountsDomain, AccountKVDomain, KVGenerationDomain, and
  CodeDomain expose typed hot readers in `DomainCfg`. `PersistentHistoryReader`
  uses that registered latest reader even when merging cold history with the hot
  tail, so archive reconstruction no longer seeds account/KV/generation/code
  baselines by calling rawdb state accessors directly.
- Latest-domain hot lifecycle hooks now include account-latest, account-KV,
  KV-generation, and CodeDomain iteration plus CodeDomain deletion. The pruning
  checker counts latest rows and validates referenced code hashes through those
  registered hooks, and the snap-mode worker scans/deletes `state-code-v1` rows
  through CodeDomain's registered lifecycle boundary instead of directly calling
  rawdb code accessors.
- CommitmentDomain and CommitmentCheckpoint now expose registered hot lifecycle
  hooks as well. The canonical range executor publishes commitment checkpoints
  through the registered checkpoint writer, the pruning checker validates
  commitment root/node/checkpoint rows through the registry, and the worker
  prunes old commitment checkpoints through the registered checkpoint
  iterator/deleter rather than calling rawdb checkpoint accessors directly.
- CodeDomain retention checks now collect hot account-envelope history through
  the registered `HistoryDomain` tx-range change iterator. The checker still
  reads CodeDomain snapshots for cold coverage, but the hot history scan used to
  decide whether a `state-code-v1` row may be pruned is no longer a direct rawdb
  changeset walk in the pruning layer.
- Canonical range execution and live CodeDomain snapshot fallback now seed their
  block -> txNum calculations through the registered `HistoryDomain` tx-range
  reader instead of calling rawdb's block-end helper directly.
- The cold history builder now selects cutoff ranges and batch-start ranges
  through the configured history domain's tx-range hooks, and the pruning
  checker uses the same registered tx-range / tx-range-change iterators for its
  retained-history accounting.
- The snapshot aggregator now discovers active AccountKVDomain families through
  the registered KV-latest iterator, and the state-domain history segment
  builder reads hot tx-ranges and range changes through the registered
  `HistoryDomain` iterators. Snapshot orchestration no longer has its own rawdb
  scans for those hot state families.
- Hot history pruning now resolves block-level row/index deletion through the
  same `HistoryDomain` registry. The worker no longer calls the rawdb
  change-set deleter directly, so hot publication, hot reads, and hot pruning
  share one domain lifecycle boundary.
- Sync import now drains contiguous buffered block runs through `InsertBlocks`
  instead of calling the single-block path for each block. The range importer
  reports the first failing index so sync can keep accurate pause diagnostics
  after a partially applied run.
- Restart-sync materialized-state rebuild now replays canonical blocks through
  the same locked `InsertBlocks` staged range importer in bounded batches,
  keeping online sync and offline rewind on one block-application surface.
- The hot history pruner now caps its prune head at canonical `Finish` stage
  progress when that watermark lags the solidified block number, so pruning
  cannot outrun the last fully finished block application.
- Canonical execution stage progress is now hash-bearing for
  `Headers/Bodies/Execution/Commitment/Finish`. Block import, fork replay, and
  restart-sync rewind write the stage watermark together with the canonical
  block hash, while generic snapshot stages can still use numeric progress.
  The hot history pruner verifies a hash-bearing `Finish` watermark against the
  current canonical block before using it as a prune cap, so pruning no longer
  trusts a detached height-only stage row.
- `applyBlock` now advances those canonical stages through a block-local
  `canonicalStagePipeline` that enforces stage order and rejects non-canonical
  snapshot/prune stages. Rewind paths use the same pipeline boundary to reset
  all canonical stages to the LCA/restart target with the target block hash.
- Canonical execution, snapshot build/prune, and hot pruning stage progress now
  cross rawdb through typed stage-progress adapters, so upper execution and
  lifecycle code no longer calls the raw stage-progress accessors directly.
- `InsertBlocks`, single-block linear import, fork replay, and restart replay
  now enter block execution through a `canonicalRangeExecutor`. The executor
  owns reusable StateDB opening, StateTxRange allocation, CommitScope reuse, and
  a per-block `canonicalBlockExecution` plan containing the planned tx range
  and stage pipeline, removing the separate `rangeState/rangeCommit/rangeTx`
  loop and the bespoke fork replay triplet.
- The block execution layer now requires that plan: `applyBlockWithPlan` no
  longer opens fallback StateDBs, derives fallback tx ranges, or creates an
  ad-hoc stage pipeline. History-enabled execution must receive the executor's
  planned `StateTxRange`, and commit/root calculation is routed through the
  `canonicalBlockExecution` object.
- The per-block state commit boundary is now owned by
  `canonicalBlockExecution.CommitState`: it runs the scoped StateDB commit,
  returns the block's latest-domain root/stat breakdown, writes optional
  `StateCommitmentCheckpoint` rows, and advances the hash-bound `Commitment`
  stage. `applyBlockWithPlan` no longer writes commitment checkpoints or stage
  progress directly.
- For scoped range execution, the account-KV latest batch flush has been
  made range-owned: `canonicalBlockExecution.CommitState` now passes a no-op
  latest flush callback through `CommitOptions`, so per-block root calculation
  does not physically write the scoped account-KV latest batch into the
  blockbuffer. The range executor closes a successful scope by flushing the
  remaining latest batch, while abort keeps already committed layers and drops
  failed active-layer writes.
- Blockbuffer batches now bind each queued operation to the active block layer
  at `Put`/`Delete` time instead of choosing the layer at `Write` time. A
  reused domain batch can therefore flush after another block has opened
  without moving the earlier block's latest-domain writes into the later layer;
  if the captured layer has already been flushed or discarded, the batch write
  fails instead of silently landing in the wrong rewind scope.
- Blockbuffer batches can also flush only the queued operations whose captured
  committed layer is at or below a solidified cutoff. `applyBlockWithPlan`
  invokes this before `postFlush`, so the async solidified-layer flusher never
  drops a blockbuffer layer before the range-owned latest batch has landed the
  corresponding account-KV latest writes.
- The scoped account-KV latest writer now keeps a pending latest view for
  batch-backed writes, and the reused `SharedDomainTx` reads through that view.
  This lets a range-owned domain tx see its own latest writes before the
  blockbuffer batch is physically flushed.
- Commitment root calculation now uses the same pending-aware latest view for
  scoped exact account-KV put/delete touches. `DomainCommitmentState` resolves
  touched latest rows through an injected latest reader, `CommitScope` supplies
  the pending-aware reader, and a missing latest-root row is initialized by
  rebuilding the physical latest base and then applying touched updates. This
  means exact put/delete block roots no longer require the account-KV latest
  batch to be physically flushed before root calculation.
- Scoped prefix deletion now uses the pending latest view as well:
  `accountKVLatestBatch.DomainDelPrefix` merges physical latest rows with
  pending batch rows instead of forcing a batch flush, and
  `RecordCommitmentMutations` expands `DomainDelPrefix` through the injected
  prefix iterator. A range can therefore put a key, keep it unflushed, and later
  delete it by prefix while still producing a root that matches a rebuild after
  the eventual flush.
- The StateDB execution read surface now attaches to the active `CommitScope`
  pending latest view. `GetAccountKV`, `GetAccountKVBatch`,
  `IterateAccountKV`, and the set/delete previous-value paths read through the
  scoped latest reader/iterator when one is active, so a later block in the same
  range can observe account-KV writes committed by an earlier block even before
  the latest batch is physically flushed.
- Journal-based temporal capture now uses the same pending-aware latest view
  when a no-read account-KV touch needs its previous value. Block-final writes
  that intentionally bypass pre-reading no longer fall back directly to the
  physical rawdb account-KV latest table, so history rows in a shared
  `CommitScope` see previous-block writes that are still only queued in the
  range-owned latest batch.
- Account latest envelopes and KV generation rows now share the same range-owned
  flat latest batch as generic account-KV latest rows. Scoped commits queue
  account latest puts/deletes and generation updates into the batch, root
  calculation reads them through the pending-aware latest reader, and range
  close/cutoff/abort flush semantics apply to all three latest-domain families
  together.
- `DomainCommitmentState` now resolves its default latest read surface through
  StateDB's typed latest reader/iterator instead of owning a rawdb latest reader.
  The StateDB read surface preserves explicit account-KV generations when the
  active latest view supports generation-aware reads, so commitment touched-key
  expansion and value lookup stay aligned with the domain transaction view.
- The StateDB/account-KV latest read/write fallback is now behind a typed
  `accountKVPhysicalLatestStore` boundary, with rawdb confined to the
  `rawdbAccountKVPhysicalLatestStore` compatibility adapter. Contract storage
  row-key derivation also reads ContractMetadata through a typed latest reader
  instead of calling rawdb directly.
- The standalone `domains.FlatStore` now follows the same boundary shape:
  account-KV latest reads, writes, deletes, prefix deletes, and iteration go
  through a typed `latestKVStore`, with rawdb isolated in
  `rawDBLatestKVStore`.
- The registered `HistoryDomain` now owns hot history pruning metadata as one
  runner: `DomainCfg` registers tx-range iteration/deletion next to hot
  `StateDomainChange` row/index hooks, and pruning worker policy is reduced to a
  prune decision callback. Full and snap modes no longer hand-compose rawdb
  `StateTxRange` scans, hot history block deletion, and tx-range deletion in the
  worker.
- Fork rewind now consumes the canonical stage watermark before discarding
  orphan layers: `switchFork` verifies that every canonical execution stage is
  present, hash-bound, and pointing at the current head before rewinding the
  stages to the LCA and replaying the winning branch. A detached or legacy
  height-only `Finish` row now rejects the fork switch instead of letting branch
  repair proceed from an unverified watermark.
- DynamicProperties' legacy flat `dp-` runtime mirror is now behind a typed
  derived-property store boundary in `core/state`. The execution-facing
  `DynamicProperties` code reads/writes only derived runtime keys through that
  adapter while rooted governance properties stay in `SystemDynamicProperty`;
  rawdb's `dp-` accessors remain the compatibility implementation, not direct
  execution-path dependencies.

Next step:

## Definition Of Done

The Erigon-style storage refactor is complete when all of the following are
true:

- Latest state is served by hot domain tables plus immutable latest files.
- Historical state is served by hot inverse indexes plus cold history
  accessors.
- Archive reads do not scan unrelated blocks, tx ranges, or full cold segments.
- Snapshot files are immutable, content-addressed, manifest-published, and
  checker-validated.
- Hot history is pruned only after cold coverage is visible.
- Commitment root computation is domain-driven and rewindable.
- Execution can run through a staged/batched pipeline.
- java-tron compatibility tests remain green.
