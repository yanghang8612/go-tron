# M8.1 Solidity / PBFT API Variants — Design

Date: 2026-04-26  
Milestone: M8.1

## Background

TRON nodes expose three read-only query variants at different confirmation depths:

| Prefix | Block reference | Trust level |
|--------|----------------|-------------|
| `/wallet/` | Current head | Fastest, unconfirmed |
| `/walletsolidity/` | Latest solidified block (`latest_solidified_block_num` DP key) | Finalized by DPoS 2/3 quorum |
| `/walletpbft/` | Latest PBFT-confirmed block (`LATEST_PBFT_BLOCK_NUM` rawdb key) | PBFT-finalized (most recent) |

For HTTP, all three prefixes exist. For gRPC, only `Wallet` (head) and `WalletSolidity` are defined in the proto (there is no `WalletPBFT` gRPC service).

## Data Model

go-tron does not maintain historical state snapshots. The solid/PBFT variants differ from the head variant only in:

1. **Block-returning endpoints** (`GetNowBlock`, `GetBlockByNum`, `GetNowBlock2`): return the block at the solid/PBFT number instead of the head.
2. **Block-bounded queries** (`GetTransactionInfoByBlockNum`, block-range queries): reject requests for block numbers above the solid/PBFT boundary.
3. **State queries** (`GetAccount`, `GetContract`, etc.): identical to head queries — state is monotonic; solid state is always a prefix of head state.

This matches java-tron behavior: `WalletSolidity.getAccount` reads from the same store as `Wallet.getAccount`.

## HTTP Design

Add two new route groups to `api.go:RegisterRoutes`:
- `RegisterSolidityRoutes(mux)` — registers `/walletsolidity/` prefixed handlers
- `RegisterPbftRoutes(mux)` — registers `/walletpbft/` prefixed handlers

Both groups reuse the existing `api.*` handler funcs for state queries. Only `getNowBlock` and `getBlockByNum` need variant-specific wrappers.

**Specialized handlers:**
- `getSolidNowBlock` — returns `bc.GetBlockByNumber(SolidifiedBlockNum())`
- `getSolidBlockByNum` — same as wallet version but 404 if `num > SolidifiedBlockNum()`
- `getPbftNowBlock` — returns `bc.GetBlockByNumber(uint64(LatestPbftBlockNum()))`; falls back to solid if PBFT not yet activated
- `getPbftBlockByNum` — same but bounded by PBFT num

**Endpoints registered under `/walletsolidity/`:**
- `getnowblock`, `getblockbynum`, `getblockbyid`, `getblockbylimitnext`
- `getaccount`, `getaccountbyid`, `getaccountresource`, `getaccountnet`
- `listwitnesses`, `getchainparameters`, `getnextmaintenancetime`
- `gettransactionbyid`, `gettransactioninfobyid`, `gettransactioninfobyblocknum`
- `getassetissuebyid`, `getassetissuebyname`, `getassetissuelist`, `getpaginatedassetissuelist`
- `getmarketorderbyid`, `getmarketordersfromaccount`, `getmarketpricebypair`
- `listexchanges`, `getdelegatedresourcev2`, `getdelegatedresourceaccountindexv2`
- `getreward`, `estimateenergy`, `triggerconstantcontract`

**Endpoints registered under `/walletpbft/`:** same list (key difference is `getnowblock` returns PBFT-confirmed block).

## gRPC Design

`internal/grpcapi/solidity.go` defines `SolidityServer` embedding `apipb.UnimplementedWalletSolidityServer`.

Methods delegate to the existing `Wallet` handlers via the shared backend, except:
- `GetNowBlock` / `GetNowBlock2` → returns `bc.GetBlockByNumber(SolidifiedBlockNum())`
- `GetBlockByNum` / `GetBlockByNum2` → bounded by solid num

All shielded methods (`GetMerkleTreeVoucherInfo`, `ScanNote*`, `IsSpend`, `ScanShielded*`, `IsShieldedTRC20*`) return `codes.Unimplemented` (not in go-tron scope).

`SolidityServer` is registered in `grpcapi.Server.Start()` alongside the existing `WalletServer`.

## Backend Interface Changes

Two new methods on `tronapi.Backend`:

```go
SolidifiedBlockNum() uint64  // latest_solidified_block_num from DynamicProperties
LatestPbftBlockNum() int64   // LATEST_PBFT_BLOCK_NUM from rawdb (-1 if not yet written)
```

`TronBackend` implements both; `stubBackend` in `api_test.go` returns 0 / -1.

## Exit Conditions

- `make test` all packages green (new tests included)
- `/walletsolidity/getnowblock` returns the solidified block, not the head
- `/walletpbft/getnowblock` returns the PBFT-confirmed block (falls back to solid if not activated)
- gRPC `WalletSolidity/GetNowBlock` returns the solidified block
- Shielded methods return `Unimplemented`
