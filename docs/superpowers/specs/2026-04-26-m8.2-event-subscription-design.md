# M8.2 Event Subscription Design

**Date**: 2026-04-26  
**Status**: Draft  
**Exit condition**: Browser can subscribe to contract logs via `eth_subscribe` over WebSocket.

---

## Goal

Add `eth_subscribe` / `eth_unsubscribe` over WebSocket to the existing JSON-RPC server. Two subscription types:
- `newHeads` — push a block header object on each new block
- `logs` — push matching log objects on each new block (same filter semantics as `eth_getLogs`)

This covers the TRON ecosystem use case where front-ends (e.g. TronWeb, ethers.js-tron) listen for smart-contract events in real time.

---

## Out of Scope

- `newPendingTransactions` (no txpool hook exists yet)
- java-tron plugin trigger system (`EventPluginLoader`)
- `internal/events/` package (the PLAN.md reference to this is aspirational; for now `FilterManager` already provides the fanout bus)
- PBFT/solidity subscription variants
- WebSocket auth, rate limiting, per-connection backpressure beyond channel buffering

## Architecture

### Transport

WebSocket upgrade is handled at the same HTTP endpoint as JSON-RPC (`/` or any path). The `ServeHTTP` handler in `api.go` currently gates on `POST` only; we intercept WebSocket upgrade requests first, before that check.

A new file `internal/jsonrpc/subscription.go` owns all WebSocket subscription logic. It keeps the existing HTTP polling path completely unchanged.

### Push format

`eth_subscription` push (EIP-758 / standard Ethereum pattern):

```json
{
  "jsonrpc": "2.0",
  "method": "eth_subscription",
  "params": {
    "subscription": "0x<id>",
    "result": <block_header or log object>
  }
}
```

This is _not_ a response to a request; it is pushed asynchronously over the WebSocket connection.

### Subscription lifecycle

1. Client sends `eth_subscribe` RPC over WebSocket.
2. Server registers a `wsSubscription` in `SubscriptionManager` and returns an ID.
3. `SubscriptionManager` holds an internal channel per subscription; it feeds events from the existing `FilterManager` fanout (reused) or directly from a block hook.
4. A per-connection goroutine reads from the channel and writes to the WebSocket (single-writer-per-connection guarantee).
5. Client sends `eth_unsubscribe` or disconnects → cleanup.

### SubscriptionManager

New struct in `subscription.go`:

```go
type SubscriptionManager struct {
    mu   sync.Mutex
    subs map[string]*wsSub   // id → subscription
}
```

`wsSub` holds:
- `kind`: `"newHeads"` or `"logs"`
- `logFilter *LogFilter` (for logs kind)
- `ch chan interface{}` (buffered, size 64)

The `SubscriptionManager` is wired into `FilterManager.fanOut`: when `fanOut` processes a block, it additionally sends to all `wsSub` channels (after applying `matchesLogFilter` for log subs, same logic already used for poll-based subs).

### Single-writer guarantee (gorilla/websocket)

gorilla/websocket requires that concurrent writes from multiple goroutines use a mutex or dedicate one writer goroutine. We use the latter: each WebSocket connection gets one write goroutine that drains a buffered `chan interface{}`.

### Dependency

Add `github.com/gorilla/websocket` to `go.mod`. No other new external dependencies.

## Test plan

- `TestWS_Subscribe_NewHeads`: connect WS, send `eth_subscribe newHeads`, inject block via hook, assert push received.
- `TestWS_Subscribe_Logs`: subscribe with address filter, inject block with matching log, assert push; inject block with non-matching log, assert no push.
- `TestWS_Unsubscribe`: subscribe, unsubscribe, inject block, assert no push.
- `TestWS_HTTP_Coexistence`: plain POST to same server still works after WS client connects.

Tests use `gorilla/websocket` over `net/http/httptest` (not bufconn, since httptest.Server handles the HTTP→WS upgrade correctly).
